# Start Agentic Verification Test — Human Walkthrough

Step through the entire Agent Workflow yourself, phase by phase. You drive each phase by pasting commands into the agent chat. This guide tells you exactly what to paste, what answers to give, and what to verify before moving on.

The experiment observer is still injected — the agent self-logs every action to `experiment-log.md` autonomously while you work.

> **Prerequisites:**
> - Read [`agentic_test_plan.md`](agentic_test_plan.md) for objectives and scoring.
> - Read [`agentic_test_runbook.md`](agentic_test_runbook.md) for per-phase detail.
> - Confirm `~/workflow/` exists (the workflow repo).
> - Optionally install mingw-w64 ahead of time: `sudo apt install -y mingw-w64`
>   (or skip it — letting the agent figure it out tests FAILURE_ROUTING).

## Phase 0 — Bootstrap (run in your terminal, not agent chat)

Run these commands in a regular terminal to set up the project:

```bash
# 0.1 — Verify cross-compiler (optional)
x86_64-w64-mingw32-gcc --version

# 0.2 — Create target directory
mkdir -p ~/hello-verify && cd ~/hello-verify
git init
git commit --allow-empty -m "Initial empty commit"

# 0.3 — Scaffold
bash ~/workflow/scripts/install.sh ~/hello-verify/
bash ~/workflow/scripts/validate-scaffold.sh ~/hello-verify/

# 0.4 — Inject experiment observer
cp ~/workflow/docs/verification/experiment_observer.md ~/hello-verify/workflow/EXPERIMENT_OBSERVER.md

cd ~/hello-verify
sed -i '/| Orchestrator state/a | Experiment observer rules              | `workflow/EXPERIMENT_OBSERVER.md` |' AGENTS.md
sed -i '/- Reference the constitution for alignment/a - After every workflow action, append a structured entry to experiment-log.md per workflow/EXPERIMENT_OBSERVER.md' workflow/BOUNDARIES.md
sed -i '/5\. Read `workflow\/ORCHESTRATOR.md`/a 6. Read `workflow/EXPERIMENT_OBSERVER.md` — experiment logging rules' CLAUDE.md

# 0.5 — Create initial experiment log
cat > experiment-log.md << 'EOF'

> Auto-generated by the executing agent per `workflow/EXPERIMENT_OBSERVER.md`.
> Append-only. Do not edit previous entries.

**Test project:** hello-verify (Windows Hello World — C/mingw-w64)
**Started:** [TO BE FILLED BY AGENT]
**Workflow repo version:** [TO BE FILLED BY AGENT]
**Tool:** [TO BE FILLED BY AGENT]

EOF

# 0.6 — Commit
git add -A
git commit -m "Scaffold + experiment observer injection"
```

**Verify before continuing:**
- [ ] `AGENTS.md` contains "Experiment observer" row

**Now open `~/hello-verify/` as a new VS Code window.** All agent chat from here happens in that window.

## Phase 2 — Compass (Project Identity)

### Paste into agent chat:

```
/compass
```

(Or use `compass.prompt.md` in Copilot agent mode.)

The agent will start an adaptive interview. It asks questions — you answer them. Use the answers below.

### Answers to give (one per question as they come up):

- **What problem does this project solve?**
  - We need to verify the Agent Workflow system works end-to-end by building a real (minimal) project through it. The project itself is a Windows "Hello, world!" executable cross-compiled from C using mingw-w64 on WSL.

- **Who is the target user?**
  - Workflow developers validating the Agent Workflow system. The .exe is a test artifact, not a product.

- **How do we know the project succeeds?**
  - A Windows `.exe` that prints exactly `Hello, world!\n` to stdout, built from C via `x86_64-w64-mingw32-gcc` on WSL, verified as a valid PE executable.

- **What are the core capabilities?**
  - Cross-compile C source to a Windows executable
  - Print "Hello, world!" to stdout
  - Exit cleanly with exit code 0

- **What is out of scope?**
  - GUI, networking, multi-file programs, Linux native builds, runtime libraries beyond the standard C library, installers, packaging

- **What principles can never be broken?**
  - Minimal code — one source file, one output binary
  - Maximum workflow coverage — exercise every phase even though the project is trivial
  - Cross-platform build — must target Windows from a Linux host

- **What security requirements apply?**
  - No external inputs, no network access, no file I/O beyond stdout. The binary is inert.

- **What testing requirements apply?**
  - Verify `.exe` is a valid PE executable (`file` command)
  - Verify stdout output is exactly `Hello, world!\n` (via Wine if available)
  - Verify process exit code is 0

### After the interview completes, verify:
- [ ] `.specify/constitution.md` exists with all 8 sections
- [ ] No `[PROJECT-SPECIFIC]` placeholders remain
- [ ] `AGENTS.md → Overview` section is filled
- [ ] `workflow/STATE.json` shows `"projectPhase": "3-define-features"`

## Phase 3 — Define Features

### Paste into agent chat:

```
/define-features
```

The agent will read the constitution and propose features. Guide it to create exactly **one feature**.

### Answers to give:

- **How many features / what features?**
  - Just one feature: `001-hello-world` — "Cross-compile Hello World to Windows executable"

- **What are the acceptance criteria?**
  - AC-1: GIVEN the source file `hello.c` exists, WHEN `x86_64-w64-mingw32-gcc` compiles it, THEN `hello.exe` is produced with exit code 0
  - AC-2: GIVEN `hello.exe` exists, WHEN inspected with `file hello.exe`, THEN output contains "PE32+" (valid Windows executable)
  - AC-3: GIVEN `hello.exe` runs (via Wine or Windows), WHEN executed, THEN stdout contains exactly "Hello, world!\n"
  - AC-4: GIVEN `hello.exe` runs, WHEN executed, THEN process exit code is 0

- **If agent proposes multiple features:** Redirect — "This is a single-feature project. Combine everything into 001-hello-world."

- **Affected areas?** `src/`, `Makefile`, `tests/`

```

```

The agent implements all tasks.

### Guidance if needed:

- **If it asks about hello.c content:**
  - Standard C, `#include <stdio.h>`, `printf("Hello, world!\n")`, `return 0`

- **If it asks about Makefile structure:**
  - `CC = x86_64-w64-mingw32-gcc`
  - `TARGET = hello.exe`
  - `SRC = src/hello.c`
  - Default target builds `$(TARGET)` from `$(SRC)`
  - `test` target runs `file $(TARGET) | grep -q "PE32"`

- **If build fails (mingw-w64 not found):** Let the agent work through FAILURE_ROUTING. If it asks: "Try `sudo apt install -y mingw-w64`"

- **If tests fail:** Let the agent retry (max 2 attempts). Only intervene if it's stuck.

## Phase 6.5 — Bug Injection (you do this manually)

**Before telling the agent to continue**, run these commands in your terminal:

```bash
cd ~/hello-verify
sed -i 's/x86_64-w64-mingw32-gcc/i686-w64-mingw32-gcc-posix-FAKE/' Makefile
git add Makefile
git commit -m "chore: update compiler target (introduces bug)"
```

This breaks the build. The compiler binary won't exist. The agent should detect this and go through the `/bug` → `/bugfix` path.

## Phase 7 — Bug Track

### Paste into agent chat:

```
Run the tests. The build should fail. Diagnose the failure,
log it via /bug, then fix it via /bugfix.
```

### What to expect:

1. Agent runs `make` → fails with "No such file or directory"
2. Agent logs BUG-001 (description, location, severity: blocking)
3. Agent identifies wrong compiler binary name in Makefile
4. Agent fixes the CC variable back to `x86_64-w64-mingw32-gcc`
5. Agent verifies build succeeds, commits with `Fix BUG-001:` prefix

### Guidance if needed:

- **If agent doesn't detect the bug:** Paste: "Run `make` and check if it succeeds."
- **If agent fixes the symptom (e.g., installs a different compiler) instead of fixing the Makefile:** Paste: "The root cause is the CC variable in the Makefile. Fix that, not the environment."
- **If agent skips /bug logging:** Paste: "Log this as BUG-001 in bugs/LOG.md before fixing. Include description, location, severity, expected, actual."

### Verify before moving on:
- [ ] `bugs/LOG.md` exists with BUG-001 entry
- [ ] Bug has all required fields
- [ ] Fix committed with `Fix BUG-001:` prefix
- [ ] Build and tests pass again
